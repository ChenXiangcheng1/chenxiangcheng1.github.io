1 RDB_MS关系型数据库架构

如何设计一个关系型数据库？

要设计一个关系型数据库，首先要将其划分成两大部分，一个是存储部分，该部分类似一个文件系统来将数据持久化到存储设备当中。那光有存储是不行的，还需要有程序实例部分来对存储进行逻辑上的管理，而程序实例部分包含将数据的逻辑关系转换成物理存储关系的1存储管理模块、优化执行效率的2缓存模块、将SQL语句进行解析的3SQL解析模块、记录操作的4日志管理模块、进行多用户管理的5权限划分模块、6灾难恢复模块、优化数据查询效率的7索引模块，以及使得数据库支持并发操作的8锁模块。

2索引模块

2.1为什么要使用索引？

因为索引能让我们避免全表扫描，提升检索效率。
全表扫描将整张表的数据全部或者分批次加载到内存当中，再搜索不仅浪费内存还花费时间。

2.2什么样的信息能成为索引？(常问)

可以是主键、唯一键以及普通键。只要是能将记录限定在一定查找范围内的具有区分性的字段

2.3索引的数据结构？

主流是B+-Tree、还有Hash结构、BitMap等

平衡二叉树或红黑树

无论是平衡二叉树还是红黑树每个节点包含一个关键字，且最多指向两棵子树，因此树的深度很大。
在检索过程中，检索深度每增加1就会发生一次磁盘IO。

m阶B-Tree

m阶B树(m>=2)特征：
1根节点至少包括两个孩子
2树中每个节点最多含有m个孩子
3除根节点和叶节点外，其他每个节点至少有ceil(m/2)个孩子。非叶子节点至少有$\lceil m/2 \rceil$个孩子节点
4所有叶子节点都位于同一层
5假设每个非终端结点中包含有n个关键字信息，其中
a) Ki (i=1…n)为关键字，且关键字按顺序升序排序K(i-1)< Ki。每个节点是至多m-1个关键字的有序表
b) 关键字的个数n必须满足:[ceil(m / 2)-1]<= n <= m-1
c) 非叶子结点的指针:P[1], P[2]，..P[M];其中P[1]指向关键字小于K[1]的子树，P[M]指向关键字大于K[M-1]的子树，其它P[i]指向关键字属于(K[i-1]，K[i])的子树

B+ Tree索引

m阶B树(m>=2)特征：
1根节点至少包括两个孩子
2树中每个节点最多含有m个孩子
3除根节点和叶节点外，其他每个节点至少有ceil(m/2)个孩子。非叶子节点至少有$\lceil m/2 \rceil$个孩子节点
4所有叶子节点都位于同一层
5假设每个非终端结点中包含有n个关键字信息，其中
a) Ki (i=1…n)为关键字，且关键字按顺序升序排序K(i-1)< Ki。每个节点是至多m-1个关键字的有序表
b) 关键字的个数 n = 非叶子节点的子树指针个数
c) 非叶子节点的子树指针P[i]，指向关键字值[K[i，K[i+1])的子树，
d) 非叶子节点仅用来索引，数据都保存在叶子节点中
f) 所有叶子节点均有一个链指针指向下一个叶子结点

B+Tree更适合用来做存储索引
B+树的磁盘读写代价更低
B+树的查询效率更加稳定
B+树更有利于对数据库的扫描，即能优化范围查询

Hash索引

是哈希索引是索引键通过哈希运算，再将运算结果的哈希值和行指针信息存放在Buckets中。

缺点：
仅仅能满足“=”，”IN”，不能使用范围查询。因为哈希值大小关系和键值的不一致，所以只能等值过滤。
无法被用来避免数据的排序操作
不能利用部分索引键做查询。而B+-Tree支持利用组合索引中的部分索引。
不能避免表扫描。由于不同索引键可能存在相同哈希值，无法从哈希索引中直接完成查询，还是要访问Bucket中的数据进行实际的比较。
遇到大量Hash值相等的情况性能并不一定就会比B-Tree索引高

BitMap位图索引

2.4密集索引和稀疏索引的区别？

表的物理排列顺序决定了密集索引，所以一个表只能有一个密集索引

密集索引文件中的每个搜索码值都对应一个索引值
稀疏索引文件只为索引码的某些值建立索引项，叶子节点只存储索引需要进一步定位

聚集索引和稀疏索引之间的主要区别在于叶子节点存储的内容不同，聚集索引存储整行数据，而稀疏索引存储行指针。使用非聚集索引需要两次IO操作

InnoDB支持聚集索引(主键索引)存储了行数据，稀疏索引只存储主键值(需要查两次 慢)
MyISAM只支持非聚集索引，存储行指针

Innodb的密集索引选取规则：

​ 若一个主键被定义，该主键则作为密集索引
​ 若没有主键被定义，该表的第一个唯一非空索引则作为密集索引
​ 若不满足以上条件，innodb内部会生成一个隐藏主键(密集索引)
​ 非主键索引存储相关键位和其对应的主键值，包含两次查找。即非主键索引的叶子节点并不存储行数据的物理地址，而是存储该行的主键值

InnoDB的数据和索引存储在一块，在ibd文件中
MyISAM的索引存储在MYI，数据存储在MYD

2.5如何定位并优化慢查询Sql？

第一步：根据慢日志定位慢查询sql

第二步：使用explain关键字等工具了解查询优化器如何执行查询

id字段：值越大执行优先级越高

type字段：其中all表示全表扫描sql语句需要优化
执行效率 system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > all

extra字段：出现以下2项意味着MYSQL根本不能使用索引,效率会受到重大影响。应尽可能对此进行优化。
Using filesort：表示MySQL会对结果使用一个外部索引排序,而不是从表里按索引次序读到相关内容。可能在内存或者磁盘上进行排序。MySQL中无法利用索引完成的排序操作称为”文件排序”
Using temporary：表示 MySQL在对查询结果排序时使用临时表。常见于排序order by和分组查询group by

key字段：表示使用的索引字段

第三步：修改sql或者尽量让sql走索引

查询优化器

mysql的查询优化器会尽可能使用严格索引来消除尽可能多的数据行

InnoDB中count(id)使用account非聚集索引索引，而不是id主键聚集索引。分析：这是因为聚集索引的叶子节点存储完整的行数据，而非聚集索引只存储主键值，因此使用非聚集索引来计算count效果更好，所以MySQL查询优化器使用account索引。

2.6联合索引的最左匹配原则的成因？

联合索引B+-Tree对各字段依次排序，因此第一个字段是绝对有序的，之后的字段在一定范围内有序。所以(col3,col2,col1)想走col1索引之前需要依靠col2和col3

联合索引的最左匹配原则 (重要)：

1.最左前缀匹配原则，mysql会一直沿着联合索引向右匹配直到遇到范围查询（>、<、between、like）就停止匹配**
比如a = 3 and b = 4 and c > 5 and d = 6如果建立(a,b,c,d)顺序的索引，d是用不到索引的，如果建立(a,b,d,c)的索引则都可以用到，a,b,d的顺序可以任意调整。
2.=和in可以乱序**，比如a = 1 and b = 2 and c = 3建立(a,b,c)索引可以任意顺序，mysql的查询优化器会帮你优化成索引可以识别的形式

2.7索引是建立得越多越好吗？

数据量小的表不需要建立索引，建立会增加额外的索引开销
数据变更需要维护索引，因此更多的索引意味着更多的维护成本
更多的索引意味着也需要更多的空间

3.1MyISAM与InnoDB关于锁方面的区别是什么

MyISAM

5.1之前的默认引擎：MyISAM，稀疏索引

  • MyISAM不支持事务，sql跑完解锁

  • MyISAM默认用的是表级锁，不支持行级锁
    当对数据 SELECT 会自动加上表级读锁 (S共享锁)
    当对数据 增删改 会自动加上表级写锁 (X排他锁)

上共享锁后支持继续上共享锁，不支持排他锁

InnoDB

5.1之后的默认引擎：Innodb，只有它支持事务和外键

  • InnoDB支持事务，commit完解锁

  • InnoDB默认用的是行级锁，也支持表级锁
    当sql用到索引则会根据索引值上锁为行级锁 (锁的细度越细代价越高)，没有索引上锁为表级锁
    当对数据 SELECT 默认为非阻塞SELECT即快照读、加锁SELECT为当前读

意向锁是一种不与行级锁冲突表级锁，表示想上锁，为数据行加共享/排他锁之前，InooDB会先获取该数据行所在在数据表的对应意向锁。
作用：上表锁时不必逐个检查行锁

MyISAM适合的场景

频繁执行全表count语句。因为MyISAM用变量保存行数，Innodb不保存行数
对数据进行增删改的频率不高，查询非常频繁。因为增删改要锁表、是非聚集索引

InnoDB适合的场景

数据增删改查都相当频繁。因为增删改只锁行

锁的分类

按锁的粒度划分，可分为表级锁、行级锁、BDB引擎页级锁
按锁级别划分，可分为共享锁、排它锁
按加锁方式划分，可分为自动锁、显式锁
按操作划分，可分为DML锁、DDL锁
按使用方式划分，可分为乐观锁 (version字段表示是否过期，先检查version再更新version++、CAS)、悲观锁 (先取锁再使用的保守策略，用数据库锁实现)

gap锁防止幻读

3.2数据库事务的四大特性 ACID

transaction 事务

ACID

  • 原子性 (atomicity [əˈtɑːmɪk])：一个事务是一个不可分割的工作单位。
  • 一致性 (consistency [kənˈsɪstənsi])：一事务必须是使数据库从一个一致性状态变到另一个一致性状态。
  • 隔离性 (isolation [ˌaɪsəˈleɪʃn])：多个并发事务执行时，它们之间是相互隔离的，一个事务的执行不应该受到其他事务的干扰。
  • 持久性 (durability [ˌdʊrəˈbɪləti])：一旦事务提交成功，它对数据库中数据的改变就应该是永久性的。

原子性：结构上，我的事务只有两种状态，全部执行成功和全部回滚 (核心)

一致性：逻辑上，要维护数据逻辑关系的正确和完整，没有矛盾

隔离性：并发上，两个事务是不能互相干扰的，每个事务看到的数据应该是一致的，不能出现并发读取数据时，读取到了其他未提交的事务的数据的情况。通过加锁解决。MySQL具体分为4种隔离级别

持久性：一旦事务提交成功，其所做的修改将会永久保存在数据库中，即使在之后的时间里发生了系统故障或断电等异常情况，数据也不应该丢失。Innodb使用redo log记录数据库修改操作保证了持久性。

3.3事务隔离级别 以及 各级别下的并发访问问题

  • 是：表示在当前隔离级别条件下，事务允许该隔离级别的操作，会发生该并发问题，线程异步执行

  • 否：不允许，不会发生，线程同步执行

事务隔离级别 并发问题 实现方式
RU read-uncommitted 会发生脏读、不可重复度、幻读问题
RC read-committed (Oracle默认) 会发生不可重复度、幻读问题 每次快照读都会创建新的read view，只能读持久化数据，不能读内存数据
RR repeatable-read (MySQL默认) 会发生幻读问题，但可以避免 MVCC多版本并发控制机制：在RR级别下，事务首次调用快照读会创建快照
当事务执行 SELECT 语句时，会根据该事务的快照和每行记录的版本号来判断是否可以读取该记录。
因此当前事务读取的数据与其他并发事务对该数据的修改是隔离的。但是，对于其他并发事务对数据进行了插入删除，则会导致幻读的出现。
serializable /ˈsɪˌriəˌlaɪzəbl/ 并发低 SELECT加共享读锁锁表，修改加排他写锁锁表

在可重复读隔离级别下，当一个事务启动时，会生成一个快照。在这个事务执行过程中，其他并发事务对于已经存在的数据进行修改时，并不会影响到当前事务的快照，因此当前事务读取的数据与其他并发事务对该数据的修改是隔离的。但是，对于当前事务还不存在的数据，其他并发事务对其进行了修改，则会导致幻读的出现。

为了解决幻读问题，MySQL 提供了两种方法：加锁和多版本并发控制。加锁方式下，在查询时会对相应的记录加锁，阻止其他事务的修改操作。多版本并发控制则是通过快照读的方式实现隔离，因此不需要显式地加锁，能够提高并发性能。

事务并发访问引起的问题 释义 如何避免
更新丢失 Rollback错误 mysql所有事务隔离级别在数据库层面上均可避免 = Rollback到记录的上一个版本
脏读 读未提交 READ-COMMITTED事务隔离级别以上可避免、只能读持久化数据不能读缓存
不可重复读 侧重于对同一事物的修改，读取与修改不隔离 REPEATABLE-READ事务隔离级别以上可避免(更好,伪MVCC避免了上锁)、使用共享读锁锁行(不建议)
幻读 侧重于新增或者删除，读取与增删不隔离 SERIALIZABLE事务隔离级别可避免、RR使用lock on share mode上next-key锁

当前读和快照读

当前读：select..lock in share mode、select..for update、update、delete、insert (加了锁的增删改查语句)

快照读(非阻塞读)：非SERIALIZABLE不加锁的非阻塞读select (串行读=当前读)

3.4RC、RR级别下的InnoDB的快照读 (非阻塞读) 如何实现

快照读实现基于伪MVCC机制(多版本并发控制)，作用读不加锁读写不冲突
undo log链记录多个事务的过程是串行化的，不属于多版本共存

  • 数据行隐藏字段：DB_TRX ID(事务ID)、DB_ROLL_PTR(回滚指针指向undo log)、DB_ROW_ID(行ID，隐藏备用主键字段)

  • undo log链：记录了数据修改之前的副本
    insert undo log链，事务提交后删除
    update undo log链，update/delete操作产生，事务回滚时需要，快照读需要，快照不涉及该日志才删除

  • read view(记录了活跃事务，与快照对应)，遵循可见行算法 (DBA范畴)
    调用快照读时，将当前事务ID与系统其他活跃事务ID做对比，若大于等于，就通过DB_ROLL_PTR指针取出undo log，直到小于活跃事务ID (越新开启的事务ID越大)，决定当前事务能看到哪个版本的数据

RC每次快照读都会创建新的read view
RR下第一条快照读会创建一个read view，将当前系统中活跃的(正在进行的)其他事务记录起来。之后调用快照读还是用的同一个read view。所以RR首次使用很快照读重要

3.5InnoDB的RR级别下如何避免幻读

表象(掩耳盗铃)：基于伪MVCC机制实现快照读

内在：RR、Serializable级别下实现了next-key锁 (record记录锁 + Gap锁)，从而避免了插删幻读

RR级别下的走主键索引或唯一索引的当前读的next-key锁机制：

  • 若where条件全部命中，则只会上记录锁
  • 若where条件部分命中或者全不命中，则会上记录锁、 (左开右闭区间) 上Gap锁中

RR级别下走非唯一索引或不走索引的当前读的next-key锁机制

  • 非唯一索引：使用主键达到更准确的区间
  • 不走索引：对所有Gap上锁=锁表

5理论范式

1NF：列不可以再分

2NF：在满足第一范式的基础上，非主键列必须直接并完全依赖主键列，不存在部分依赖

3NF(=BCNF)：在满足第二范式的基础上，非主键列之间不存在依赖，即没有冗余即不存在传递依赖

有时候我们会有反范式的设计，为了提高数据库查询的性能

外键一般在所有的表创建完成后，用alter table添加外键类型约束的方式创建，好处是不用考虑创建表的sql顺序

外键的级联操作

互联网行业应用不推荐使用外键，因为如果用户量大，并发度高，为此数据库服务器很容易成为性能瓶颈，尤其受IO能力限制，且不能轻易地水平扩展。
外键需要额外的维护，更消耗资源，还会因为需要请求其他表，对其他表加锁而容易出现死锁情况。

若是把数据一致性的控制放到事务中，也即让应用服务器承担此部分的压力，而引用服务器一般都是可以做到轻松地水平的伸缩；

级联更新和级联删除

如果父表中的记录被删除，则子表中对应的记录自动被删除

对子表添加级联

  • on update：是指在主键表中被参考字段的一条记录的值更新时，进行某种操作

  • on delete：是指在主键表中被参考字段的一条记录的值被删除时，进行某种操作

on update 和 on delete 后面可以跟的参数有四个

  • no action 表示 不做任何操作，
  • set null 表示在外键表中将相应字段设置为null
  • set default 表示设置为默认值
  • cascade 表示级联操作，就是说，如果主键表中被参考字段更新，外键表中也更新，主键表中的记录被删除，外键表中改行也相应删除

表间关系

多对多关系

多对多关系实际上是通过两个一对多关系实现的，即通过一个中间表，关联两个一对多关系，就形成了多对多关系
多对多关系的外键则是在中间表中的两个字段上设置两个外键

一对一关系

一对一关系是指，一个表的记录对应到另一个表的唯一一个记录。目的是把经常读取和不经常读取的字段分开，以获得更高的性能。

例如，把一个大的用户表分拆为用户基本信息表user_info和用户详细信息表user_profiles，大部分时候，只需要查询user_info表，并不需要查询user_profiles表，这样就提高了查询速度。

为什么会提高查询速度？

  • 避免null，性能更快。一个表会有Null存在，两个表，另一个表中直接没有记录了
  • 避免每次访问数据库都要连带操作经常不需要信息
